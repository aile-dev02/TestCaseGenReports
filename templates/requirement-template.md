# テンプレートの使い方

## コピーして使う手順

1. このファイルを `master/requirements/` にコピーする
2. ファイル名を `REQ-{カテゴリ}-{連番3桁}.md` に変更する（例: `REQ-PAYMENT-001.md`）
3. FrontMatter の各フィールドを埋める
4. 本文に背景・目的を記述し、`# 受入基準` を箇条書きで列挙する
5. この説明セクション（`---` 以下）を削除する

---

## フィールド説明

| フィールド | 必須 | 説明 | 設定例 |
|:----------|:----:|:-----|:------|
| `id` | ✅ | 一意の識別子。`REQ-{カテゴリ}-{3桁番号}` 形式 | `REQ-LOGIN-001` |
| `タイトル` | ✅ | 要件名を端的に表すタイトル | `ユーザーログイン機能` |
| `説明` | 任意 | トレーサビリティマトリクスに表示される一行説明 | `ユーザーがID/PWでログインできること` |
| `カテゴリ` | ✅ | 機能カテゴリ（英大文字推奨） | `LOGIN`, `USER`, `PAYMENT` |
| `優先度` | ✅ | ビジネス上の優先度 | `high` / `medium` / `low` |
| `ステータス` | ✅ | 要件のライフサイクル状態。下の表を参照 | `approved` |

---

## ステータスの選び方

| 値 | 意味 | 使うタイミング |
|:--|:-----|:-------------|
| `draft` | ドラフト | 要件をレビュー前・検討中の段階 |
| `approved` | 承認済み | ステークホルダーの承認が得られた要件 |
| `deprecated` | 廃止 | 仕様変更等で不要になった要件（削除せず残す） |

---

## 受入基準の書き方

受入基準は「〜できること」「〜が表示されること」のように **検証可能な状態** で記述してください。

```
良い例:
  - 正しい認証情報でログインでき、ダッシュボードへ遷移すること
  - 誤った認証情報でエラーメッセージが表示されること

悪い例:
  - ログインがうまく動くこと  ← 曖昧
  - セキュリティを考慮すること ← 検証不可
```

---

## 記述例（完成形）

```markdown
---
id: REQ-LOGIN-001
タイトル: ユーザーログイン機能
説明: ユーザーが登録済みのIDとパスワードでシステムにログインできること
カテゴリ: LOGIN
優先度: high
ステータス: approved
---

ユーザーは登録済みのユーザーIDとパスワードを使用してシステムにログインできること。
認証に失敗した場合は明確なエラーメッセージを表示し、連続失敗時はアカウントをロックすること。

# 受入基準

- 正しい認証情報でログインでき、ダッシュボードへ遷移すること
- 誤った認証情報でエラーメッセージが表示されること
- 連続ログイン失敗（5回）でアカウントがロックされること
- パスワードはハッシュ化されてDBに保存されること
```

## テストケースとの紐づけ方

要件を作成したら、対応するテストケースの `要件ID` フィールドに設定します。

```yaml
# master/testcases/AUTH-001.md
---
id: AUTH-001
タイトル: 正常ログイン
要件ID: REQ-LOGIN-001   ← ここに要件IDを設定
...
---
```

トレーサビリティマトリクス（`npm run generate:traceability`）で、
要件とテストケースの対応が自動的にレポートに反映されます。
